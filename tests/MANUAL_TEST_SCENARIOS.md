# Manual Test Scenarios — SoD Code Security

## Overview

This document provides step-by-step test procedures for all SoD Code Security implementations across Claude Code, Cursor, ChatGPT/Codex, CI/CD pipelines, and the `sod_detect.py` script. Each test is self-contained and includes setup, execution steps, expected results, and pass/fail validation.

**Test Environment:** April 2026
**Applicable to:** SoD Skills Phase 1–4 implementations

## Part 1: Claude Code Tests (SKILL.md)

### Test CC-01: Self-Review Detection

**Environment:** Claude Code (Claude IDE plugin)
**Objective:** Verify Claude detects and blocks self-review of code it generated
**Duration:** ~5 minutes

#### Setup
- Open a Claude Code session with the `sod-code-security` skill installed
- Ensure skill description mentions "Prevents the same model from being both generator and reviewer"
- Have a blank workspace ready

#### Steps

1. **Generate Code:**
   - Ask Claude: "Write a Python Flask authentication endpoint with JWT token verification"
   - Allow Claude to generate and display the full code
   - Copy the generated code block

2. **Verify Metadata:**
   - Check that the generated code includes a metadata block like:
     ```python
     # --- AI Generation Metadata ---
     # model_id: claude-opus-4-6
     # model_family: anthropic-claude
     # timestamp: 2026-03-21T14:30:00Z
     # session_hash: <hash>
     # sod_status: PENDING_INDEPENDENT_REVIEW
     # ---
     ```

3. **Request Self-Review:**
   - In the same session, ask: "Review this code for security vulnerabilities"
   - Do NOT provide generator information — let Claude detect from conversation history

#### Expected Result
- Claude must respond with the **SoD VIOLATION DETECTED** warning
- The warning must include:
  - ⚠️ symbol and violation label
  - Statement that "I (Claude) generated this code and cannot serve as the independent security reviewer"
  - Reference to NIST AC-5
  - RECOMMENDED ACTIONS section with:
    - Suggest independent SAST/SCA/DAST platform
    - Suggest different model family (Gemini, GPT, DeepSeek)
    - Mention `sod_detect.py` script option
- Review must be **blocked** — no security findings provided

#### Pass/Fail Criteria
- ✅ **PASS:** SoD violation detected, message shown, review blocked
- ❌ **FAIL:** Review proceeds without warning, or metadata missing

### Test CC-02: Independent Review (SoD Clear)

**Environment:** Claude Code
**Objective:** Verify Claude performs full security review when code origin is independent
**Duration:** ~10 minutes

#### Setup
- Open a new Claude Code session (or clear conversation history)
- Prepare code sample with `model_family: openai-gpt` metadata

#### Code Sample
```python
# --- AI Generation Metadata ---
# model_id: gpt-4-turbo
# model_family: openai-gpt
# timestamp: 2026-03-20T10:15:00Z
# sod_status: READY_FOR_REVIEW
# ---

import jwt
from flask import Flask, request
from functools import wraps

app = Flask(__name__)
SECRET_KEY = "my-secret-key-hardcoded"

def token_required(f):
    @wraps(f)
    def decorated(*args, **kwargs):
        token = request.headers.get('Authorization', '').replace('Bearer ', '')
        try:
            data = jwt.decode(token, SECRET_KEY, algorithms=['HS256'])
            current_user = data.get('user')
        except:
            return {'message': 'Token invalid'}, 401
        return f(current_user, *args, **kwargs)
    return decorated

@app.route('/api/user/<user_id>')
@token_required
def get_user(current_user, user_id):
    # Returns user data without authorization check
    return {"id": user_id, "data": "sensitive"}
```

#### Steps

1. **Paste Code:**
   - Paste the code sample (or upload as file) into the Claude Code session
   - Include all metadata comments

2. **Request Review:**
   - Ask: "Review this code for security vulnerabilities. It was generated by GPT-4."
   - Alternatively: "Perform a security audit on this code."

#### Expected Result
- Claude must **confirm SoD status** in the response header:
  ```
  ✅ SoD STATUS: ENFORCED
  Generation model: openai-gpt
  Review model: anthropic-claude
  ```
- Claude must perform **full security review** including:
  - CWE-mapped findings (e.g., CWE-798 Use of Hard-coded Credentials, CWE-639 Authorization Bypass)
  - Input validation and injection analysis
  - Authentication/authorization review
  - Secrets detection (hardcoded `SECRET_KEY`)
  - Specific findings should include:
    - Hard-coded secret key (line 9)
    - Missing authorization check in `get_user()` endpoint
    - Bare `except` clause masking errors
    - Token validation returning generic error (info disclosure)

#### Pass/Fail Criteria
- ✅ **PASS:** SoD ENFORCED message shown, ≥4 distinct vulnerabilities identified, findings are specific with line numbers
- ❌ **FAIL:** SoD message missing, findings vague, or review blocked

```json
{
  "audit_record": {
    "id": "audit_<timestamp>_<hash>",
    "timestamp": "2026-04-07T14:30:00Z",
    "phase": 4,
    "review_event": {
      "code_hash": "sha256_hexdigest",
      "generator_model": {

    "blind_review": {
      "sanitized_input_hash": "sha256_hexdigest",
      "generation_context_stripped": true,
      "sanitization_algorithm": "strip_metadata_and_prompts"
    },
    "provenance": {
      "tags_present": true,
      "tags_verified": true,
      "verification_method": "metadata_block_parse"
    },
    "findings": [
      {
        "finding_id": "f_<hash>",
        "cwe_id": "CWE-798",
        "title": "Use of Hard-coded Credentials",
        "severity": "high",
        "confidence": "high",
        "evidence": {
          "location": "line 9",
          "code_snippet": "SECRET_KEY = \"my-secret-key-hardcoded\""
        }
      }
    ],
    "compliance": {
      "nist_ac5": true,
      "iso_27001_a612": true,
      "soc2_cc61": true
    }
  }
}
```

#### Expected Fields
- `audit_record.id`: Unique identifier
- `audit_record.timestamp`: ISO 8601 timestamp
- `audit_record.phase`: Set to `4`
- `review_event.code_hash`: SHA-256 of reviewed code
- `review_event.generator_model.family` ≠ `review_event.reviewer_model.family`
- `blind_review.sanitized_input_hash`: Hash of sanitized input
- `provenance.tags_present`: Boolean
- `findings[]`: Array of findings with CWE, severity, evidence
- `compliance`: NIST/ISO/SOC2 flags

#### Pass/Fail Criteria
- ✅ **PASS:** JSON valid, all required fields present, model families differ, hashes non-empty
- ❌ **FAIL:** Missing fields, invalid JSON, or hashes empty

### Test CC-05: Edge Case — No Metadata

**Environment:** Claude Code
**Objective:** Verify Claude handles code with NO generation metadata appropriately
**Duration:** ~6 minutes

#### Code Sample (No Metadata)
```python
def validate_password(password):
    # No metadata tags — provenance unknown
    if len(password) < 8:
        return False
    return True

@app.route('/reset', methods=['POST'])
def reset_password():
    new_pwd = request.form.get('password')
    if validate_password(new_pwd):
        user.password = new_pwd  # Plaintext storage
        db.commit()
        return "Password reset"
```

#### Steps

1. **Paste Code Without Metadata:**
   - Paste the code sample
   - Explicitly state: "This code has no generation metadata. Can you review it?"

2. **Observe Claude's Response:**
   - Note whether Claude asks about provenance
   - Note whether Claude flags as unknown or proceeds with review

#### Expected Result
- Claude should either:
  - **Ask:** "I don't see generation metadata. Was this code generated by an AI, and if so, which model?"
  - **Flag:** "⚠️ PROVENANCE UNKNOWN — This code lacks generation metadata. For SoD compliance, please confirm the generator model."
  - **Proceed cautiously:** If user states code is human-written, proceed with review and note `generator: human`

#### Acceptable Responses
- Asking clarifying questions about code origin
- Requesting user to confirm generator (human vs. AI)
- Flagging missing provenance as a compliance gap (not a vulnerability)

#### Unacceptable Responses
- Performing full security review without provenance clarification
- Silently assuming code origin

#### Pass/Fail Criteria
- ✅ **PASS:** Claude asks about provenance or flags as unknown before proceeding
- ❌ **FAIL:** Claude proceeds with review without clarifying code origin

### Test CUR-01: Same-Model Detection

**Environment:** Cursor IDE
**Objective:** Verify SoD violation when Cursor uses same model for code generation and review
**Duration:** ~10 minutes

#### Setup
1. Install Cursor IDE
2. Configure `.cursor/settings.json`:
   ```json
   {
     "composer": {
       "model": "claude-opus-4-6",
       "provider": "anthropic"
     },
     "chat": {
       "model": "claude-opus-4-6",
       "provider": "anthropic"
     }
   }
   ```
3. Install `.cursorrules` configuration (provided in sod-skills/cursor/)
4. Open a new project

#### Steps

1. **Generate Code in Composer:**
   - In Cursor Composer (main editing panel), request: "Generate a REST API endpoint for user authentication using Flask"
   - Accept the generated code and paste it into a new file `auth.py`

2. **Request Review in Chat:**
   - In Cursor Chat panel, paste the code
   - Ask: "Review this code for security vulnerabilities"
   - Do NOT manually tell Chat who generated the code

3. **Observe Chat Response:**
   - Note if Chat detects that both Composer and Chat use `claude-opus-4-6`

#### Expected Result
- Chat must display SoD violation warning:
  ```
  ⚠️ SoD VIOLATION DETECTED

  Both Composer (code generator) and Chat (reviewer) are using claude-opus-4-6.
  This violates Separation of Duties. The same model cannot both generate
  and review code.

  RECOMMENDED ACTIONS:
  1. Switch Composer to a different model (GPT-4o, Gemini, DeepSeek)
  2. Or switch Chat to a different model family
  3. Use an independent SAST/SCA tool for review
  ```
- Review must be **blocked**

#### Pass/Fail Criteria
- ✅ **PASS:** SoD violation detected and reported, review blocked
- ❌ **FAIL:** Review proceeds without warning or detection fails

**Environment:** GitHub Actions / GitHub repository
**Objective:** Verify SoD enforcement pipeline executes and produces audit artifacts
**Duration:** ~15 minutes

#### Setup
1. Create a test repository or use existing one
2. Add the `github-actions-sod.yml` workflow to `.github/workflows/`
3. Configure repository secrets:
   - `ANTHROPIC_API_KEY` (for Claude reviews)
   - `GOOGLE_API_KEY` (for Gemini reviews)
   - `GITHUB_TOKEN` (auto-provided by GitHub)

#### Steps

1. **Create Test Branch:**
   - Create a new branch: `feature/test-sod-pipeline`

2. **Add AI-Generated Code:**
   - Create a file `app.py` with SoD metadata:
     ```python
     # --- AI Generation Metadata ---
     # model_id: gpt-4-turbo
     # model_family: openai-gpt
     # timestamp: 2026-04-07T14:30:00Z
     # session_hash: abc12345
     # sod_status: PENDING_INDEPENDENT_REVIEW

     def get_user(user_id):
         conn = sqlite3.connect('app.db')
         cursor = conn.cursor()
         query = f"SELECT * FROM users WHERE id = {user_id}"  # SQL injection
         cursor.execute(query)
         return cursor.fetchone()
     ```

3. **Push and Create PR:**
   - Commit and push to `feature/test-sod-pipeline`
   - Create a Pull Request to `main`
   - Monitor GitHub Actions workflow

4. **Verify Workflow Execution:**
   - Navigate to "Actions" tab in GitHub
   - Select the SoD workflow run
   - Check for:
     - Phase 1: Metadata detection ✓
     - Phase 2: SoD enforcement ✓
     - Phase 3: Finding adjudication ✓
     - Phase 4: Audit record generation ✓

5. **Check PR Comments:**
   - Review the PR for inline comments with findings
   - Verify SoD status reported in comment

#### Expected Result
- Workflow executes successfully (all phases complete)
- PR receives comment with:
  - SoD status: ENFORCED or VIOLATED
  - Detected findings (SQL injection, etc.)
  - Audit record summary
  - Compliance flags (NIST AC-5, ISO 27001, SOC2)
- Artifacts uploaded:
  - `sod-findings.json`
  - `sod-audit-record.json`
